# 排障 Skill

## 目的

把模糊的技术问题，推进成有范围、有证据、有结论、有闭环的支持记录。

## 标准流程

```text
现象 -> 范围 -> 假设 -> 验证 -> 结论 -> 同步 -> 闭环
```

## 输入

- 准确现象或报错
- 预期行为
- 受影响和未受影响范围
- 时间线和近期变更
- 环境信息
- 日志、命令输出，必要时截图
- Owner 和下一次同步时间

## 步骤

1. 现象：用一句准确的话重写问题。
2. 范围：确认谁受影响、谁不受影响。
3. 假设：列 2-4 个最可能原因，按证据和影响范围排序。
4. 验证：先跑只读检查；记录命令、主机、时间、输出和解释。
5. 结论：说明已确认什么、排除了什么、还不知道什么。
6. 同步：用短更新说明状态、影响、证据、下一步和 Owner。
7. 闭环：只有完成验证、确认、剩余风险说明和经验沉淀后才关闭。

## 收集事实规则

- 优先保存文本日志和命令输出，截图只作补充。
- 记录时间戳和时区。
- 网络问题必须把源地址、目标地址、协议和端口写清楚。
- 对比正常案例和异常案例。
- 保留原始错误文本。
- 外发前脱敏。

## 缩小范围的问题

- 是一个用户还是所有用户？
- 是一台主机还是多台主机？
- 是一个网段、站点、区域，还是全局？
- 是一个版本还是所有版本？
- 是单个功能还是整个服务？
- 是否发生在某次变更之后？

## 进展同步格式

```text
状态：
影响：
证据：
当前判断：
下一步验证：
Owner / ETA：
需要：
```

## 闭环标准

- 预期行为已经验证。
- 客户或干系人确认已经记录。
- 根因或当前最佳判断已经记录。
- 剩余风险有跟进 Owner。
- 可复用经验已经进入模板、文档、FAQ 或脚本待办。

## 避免

- 只说“应该好了”，但没有验证。
- 只说“网络问题”，但没有源、目标、协议和证据。
- 只说“没看到日志”，但不写检查了哪里。
- 一次改多个变量。
- 没有证据摘要就升级。
